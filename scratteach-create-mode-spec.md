# Scratteach「いっしょにつくるモード」実装仕様書

> 対象：Claude Code  
> 作成日：2026-05-22

---

## 1. 概要

現行の「質問モード」に加えて「いっしょにつくるモード」を追加する。
ユーザーがゲームを最初から設計・制作できるモード。

---

## 2. モード比較

| 項目 | 質問モード | いっしょにつくるモード |
|------|------|------|
| 目的 | ピンポイントの疑問を解決 | ゲームを最初から設計・制作 |
| 会話スタイル | 質問→即答 | 対話しながら仕様を固める |
| メインカラー | Orange `#FF8C00` | Sky Blue `#5BB8F5` |
| サブカラー | Soft Peach `#FFCBA4` | Soft White Blue `#E8F4FD` |
| 背景グラデーション | オレンジ→ピーチ | 水色→白青 |
| ボタンカラー | オレンジ | 水色 |
| スクラの表情 | おっとり・穏やか（通常） | 少し前のめり・考えてる顔 |
| ブロック表示 | 部分的なブロック | スプライトごとに分割した完成形 |
| 画像添付 | ✅ あり | ✅ あり |

---

## 3. モード切り替えUI

### トップ画面またはサイドメニューに2つのボタンを配置

```
┌─────────────────┐  ┌─────────────────────┐
│   💬 質問モード   │  │ 🛠 いっしょにつくる  │
│  （オレンジ）     │  │    （水色）           │
└─────────────────┘  └─────────────────────┘
```

- モード切り替え時は画面全体のカラーテーマが切り替わる
- ヘッダーにも現在のモードを表示する
- モードをまたいで会話履歴は別セッションで管理する

## 4. いっしょにつくるモードの会話フロー

### Step 1：ゲームの種類を把握

ユーザーが「テトリスみたいなのを作って」など大きな要望を言ってきたら：

```
「では一緒に仕様を決めていきましょう！
いくつか質問させてください。」
```

### Step 2：必要項目を洗い出して1つずつ質問

AIが必要な仕様項目を内部でリストアップし、1問ずつ順番に質問する。

**質問例（テトリスの場合）**
```
・フィールドのサイズは？（標準は10×20）
・落下速度は？（固定 or レベルで変化）
・レベルアップの条件は？
・スコアの計算方法は？
・ネクストブロックの表示は？
・ゲームオーバーの条件は？
・BGM・効果音は必要？
```

### Step 3：再思考・追加質問

全項目の回答が終わったら、AIが一度全体を俯瞰して不足点を確認する。

```
「ありがとうございます。内容を確認しました。
もう少し確認させてください。
・ポーズ機能は必要ですか？
・ハイスコアの保存は必要ですか？」
```

### Step 4：仕様サマリーの提示とユーザー承認

```
【決定した仕様】
・フィールド：10×20
・落下速度：3段階（レベルで変化）
・スコア：1列消すごとに100点、2列で250点
・ゲームオーバー：最上段に到達
・ネクスト表示：あり
・ポーズ：あり
・ハイスコア保存：なし

この仕様でブロックを生成しますか？
［はい、作って！］　［仕様を修正する］
```

### Step 5：ブロック生成

承認後、スプライトごとに分割してブロックを生成・表示する。

---

## 5. ブロックの分割表示

大きなゲームになるとブロック量が膨大になるため、スプライト単位で分けて表示する。

```
【フィールド管理】のブロック　　→ 折りたたみ表示
【プレイヤー操作】のブロック　　→ 折りたたみ表示
【スコア管理】のブロック　　　　→ 折りたたみ表示
【ゲームオーバー処理】のブロック→ 折りたたみ表示
```

各セクションに「このブロックは◯◯スプライトに貼ってください」という案内を表示する。

## 6. 途中保存・再開機能

### 保存タイミング
- 仕様決めの途中（各質問への回答ごと）
- 仕様確定時
- ブロック生成後

### 保存先
IndexedDB（既存のDB構造に追加）

### 保存データ構造

```json
{
  "session_id": "uuid",
  "mode": "create",
  "status": "planning | confirmed | generated",
  "game_type": "テトリス",
  "spec": {
    "field": "10×20",
    "speed": "3段階",
    ...
  },
  "conversation": [...],
  "created_at": "2026-05-22",
  "updated_at": "2026-05-22"
}
```

### 再開UI

トップ画面に「前回の続きから再開する」ボタンを表示する。

## 7. 仕様変更への対応

ブロック生成後も仕様変更できるようにする。

```
生成されたブロックの上部に
［仕様を修正する］ボタンを常時表示
　↓
修正したい項目だけを再質問
　↓
変更箇所のブロックのみ差し替え or 全体再生成を選択
```

## 8. 画像添付機能（両モード共通）

### 概要

質問モード・いっしょにつくるモード両方でチャット入力時に画像を添付できる。
Gemini APIのVision機能を使用。

### 主な用途

| 場面 | 活用方法 |
|------|------|
| エラー発生時 | エラー画面を撮影して送る→原因と修正ブロックを提示 |
| ブロック確認 | 作ったブロックを撮影→合っているか確認 |
| 参考画像 | 作りたいゲームの参考画像を送る→仕様提案に活用 |
| 制作途中 | 途中のブロックを見せて「次は何を追加できる？」 |

### UI

```
チャット入力欄の左に📎ボタンを追加
　↓
スマホ：カメラ起動 or アルバム選択
PC　　：ファイル選択ダイアログ
　↓
サムネイル表示（送信前に確認できる）
　↓
テキストと一緒に送信
```

### API実装

```javascript
// Gemini APIに画像を添付する場合
{
  role: "user",
  parts: [
    { text: "ユーザーのテキスト" },
    {
      inline_data: {
        mime_type: "image/jpeg", // or "image/png"
        data: "base64エンコードした画像データ"
      }
    }
  ]
}
```

### 対応フォーマット
- JPEG / PNG / GIF / WebP
- 最大サイズ：5MB

## 9. Scratchの制約をAIに徹底させるプロンプト設計

### 最重要事項

いっしょにつくるモードでは仕様設計の段階から「Scratchで実現できるか」を常に意識させる。

### システムプロンプトに追加する内容

```
【Scratchの制約を厳守すること】

以下のルールを必ず守ること：

1. 提案するブロックは必ずScratch 3.0の公式ブロック一覧に存在するもののみ使用する
2. 一般的なプログラミング概念をScratchに置き換えて考えること
   - 配列 → リスト
   - 関数（戻り値あり）→ ブロックを作る（戻り値なし）
   - 非同期処理 → 複数スプライトの並列実行
   - クラス・オブジェクト → スプライトのクローン
3. Scratchで実現不可能な仕様はその場で指摘し、代替案を提示すること
4. 永続保存が必要な場合はクラウド変数（要Scratchアカウント）の制限を説明すること
5. 物理演算は自前実装が必要なことを考慮した仕様にすること

【Scratchの主な制限】
・データ構造：変数・リストのみ（配列・オブジェクト・辞書なし）
・関数：「ブロックを作る」のみ（戻り値なし）
・永続保存：クラウド変数のみ（要アカウント・数値のみ）
・物理演算：なし（自前実装）
・文字列操作：基本的なもののみ
```

## 10. 難易度の自動調整（Phase2以降）

会話の中からユーザーレベルを判断して提案の複雑さを調整する。

| レベル | 判断基準 | 提案の方針 |
|------|------|------|
| 初心者 | 基本的な質問・シンプルなゲーム希望 | ブロック数を最小限に抑えたシンプルな仕様を提案 |
| 中級者 | 具体的な仕様を自分で言える | より凝った仕様・エフェクトなども提案 |
